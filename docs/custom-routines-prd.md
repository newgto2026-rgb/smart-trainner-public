# 커스텀 루틴 PRD

## 1. 문제 정의
현재 Smart Trainner는 기본 루틴을 추천하고 `1일차 -> N일차`로 진행하는 흐름을 제공하지만, 사용자가 직접 `1일차에는 어떤 운동`, `2일차에는 어떤 운동`처럼 루틴을 꾸밀 수 없다. 커스텀 루틴이 추가될 때 가장 큰 제품 리스크는 사용자가 `기본 루틴`, `내 루틴`, `추천 미리보기`, `현재 진행 루틴`을 혼동하는 것이다.

## 2. 목표
- 사용자가 기존 운동 카탈로그에서 운동을 골라 1~7일차 루틴을 직접 만든다.
- 각 일차는 운동 순서, 세트, 반복 범위, 시간, 휴식 시간을 가진다.
- 저장한 루틴은 `내 루틴`으로 표시되고 앱 재시작 후에도 유지된다.
- 기본 루틴은 `기본 루틴`, 사용자 루틴은 `내 루틴`으로 명확히 구분한다.
- Plan 탭은 현재 루틴과 일정 확인에 집중하고, 루틴 목록/복사/추천 선택은 별도 선택 팝업에서 처리한다.
- 저장과 시작을 분리해, 루틴을 저장하는 것만으로 현재 진행 루틴이 바뀌지 않는다.
- 커스텀 루틴도 홈의 다음 운동, 운동 기록, 일차 완료, 다음 일차 이동 흐름을 그대로 사용한다.
- 루틴에 넣기 전에도 후보 운동의 방법과 이미지를 바로 확인할 수 있게 한다.
- 커스텀 루틴의 주요 부위는 사용자가 일차별로 선택한 경우에만 표시한다.
- 주요 부위를 선택한 일차에서는 해당 부위에 맞는 운동 후보만 보여준다.
- 커스텀 루틴에는 앱이 운동 시간을 추정해 표시하지 않는다.
- 저장한 내 루틴은 나중에 다시 수정할 수 있다.
- 4일 이상 커스텀 루틴의 진행 구성은 말줄임으로 숨기지 않고 설정된 모든 일차를 보여준다.
- 현재 선택된 내 루틴은 현재 루틴 카드에서도 바로 수정할 수 있다.
- 홈의 다음 운동 카드는 `오늘 운동 계획`으로 표현하고 해당 일차의 전체 운동을 보여준다.

## 3. 비목표
- 자유 텍스트 운동 생성.
- AI 자동 루틴 생성.
- 고급 주기화, 디로드, RPE 기반 처방.
- 영상/사진 기반 자세 분석.
- 서버 동기화를 앱의 필수 동작으로 만드는 것.

## 4. 10라운드 에이전트 토론
참여 관점: 기획 에이전트, 디자이너 에이전트, Android 아키텍트 에이전트, 서버 에이전트, Codex 조정자.

### Round 1: 문제의 중심
- 기획: 사용자는 직접 루틴을 만들고 싶지만, 앱이 추천한 루틴과 내가 만든 루틴이 섞이면 불안해진다.
- 디자인: 화면 첫 신호가 중요하다. `기본 루틴`과 `내 루틴` 배지가 없으면 사용자는 무엇을 편집하는지 헷갈린다.
- 아키텍트: UI 기능이 아니라 도메인 엔티티다. 모델과 repository 계약이 바뀌어야 한다.
- 서버: 세션 소유 custom routine API가 필요하다.
- 합의: 커스텀 루틴은 제품/도메인/서버 계약이 모두 필요한 기능이다.

### Round 2: MVP 범위
- 기획: 루틴명, 1~7일차, 운동 추가/삭제/순서 변경, 세트/반복/시간이면 충분하다.
- 디자인: 빈 상태와 저장 후 복귀가 없으면 완성감을 잃는다.
- 아키텍트: 복잡한 편집 기능은 충돌을 늘린다. MVP는 안정적 저장과 시작을 우선한다.
- 서버: nested partial patch보다 days 전체 교체가 명확하다.
- 합의: MVP는 카탈로그 기반 커스텀 루틴 저장/시작/삭제와 기본 편집으로 제한한다.

### Round 3: 기본 루틴과 내 루틴 구분
- 기획: 추천 루틴과 내 루틴을 병렬로 제공하되 대체 관계로 보이면 안 된다.
- 디자인: 한 리스트에 섞지 말고 섹션과 배지로 분리해야 한다.
- 아키텍트: 모델에 `RoutineSource.SYSTEM/CUSTOM`이 있어야 추천 알고리즘이 사용자 루틴을 후보로 보지 않는다.
- 서버: custom routine은 별도 API와 별도 선택 상태를 가진다.
- 합의: source 구분은 모델, UI, API 모두에 들어간다.

### Round 4: 진행 단위
- 기획: 사용자는 요일보다 `1일차/2일차` 흐름을 이해해야 한다.
- 디자인: 편집 화면도 요일을 전면에 두지 말아야 한다.
- 아키텍트: 기존 routine progress가 day index 기반이므로 재사용 가능하다.
- 서버: API도 `day` 숫자를 사용하면 Android와 맞다.
- 합의: 커스텀 루틴도 요일 고정이 아닌 `1일차 -> N일차` 순환이다.

### Round 5: 저장과 시작
- 기획: 저장만 했는데 현재 루틴이 바뀌면 사용자는 당황한다.
- 디자인: `저장`과 `이 루틴 시작` 버튼을 분리해야 한다.
- 아키텍트: active routine progress 초기화는 start action에서만 수행한다.
- 서버: select endpoint도 명시 액션으로 둔다.
- 합의: 저장은 목록에 추가, 시작은 현재 루틴 변경과 day 1 재시작이다.

### Round 6: 기본 루틴 편집
- 기획: 사용자는 기본 루틴을 자기 식으로 바꾸고 싶을 수 있다.
- 디자인: 기본 루틴에 `편집`을 바로 노출하면 원본이 훼손되는 느낌이 든다.
- 아키텍트: system seed는 immutable로 두고 복사본을 custom으로 저장해야 한다.
- 서버: 생성 request는 custom routine으로만 받는다.
- 합의: 기본 루틴은 `복사해서 수정`만 허용하고 원본은 편집하지 않는다.

### Round 7: 후보 운동과 중복
- 기획: 사용자는 “이미 넣은 운동을 왜 또 고를 수 있지?”에서 헷갈린다.
- 디자인: 같은 일차에 추가한 운동은 후보 목록에서 사라져야 이동/삭제 결과가 분명하다.
- 아키텍트: 기록 id는 slot index를 포함해 방어적으로 안전하게 두되, MVP 빌더는 같은 일차 중복 추가를 막는다.
- 서버: API는 order를 보존하지만 Android 빌더 정책은 per-day unique exercise로 둔다.
- 합의: 커스텀 빌더에서는 같은 일차에 동일 운동을 중복 추가하지 않는다. 이미 추가된 운동은 후보에서 숨긴다.

### Round 8: 저장소
- 기획: 앱 재시작 후 내 루틴이 남아야 한다.
- 디자인: 저장 실패/빈 상태를 사용자가 이해해야 한다.
- 아키텍트: DataStore JSON은 빠르지만 구조가 커진다. Room v4 테이블이 장기적으로 안전하다.
- 서버: JSON file store는 서버 MVP에 맞지만 앱 로컬은 Room이 적합하다.
- 합의: Android는 Room에 custom routine을 저장하고, 진행 상태는 기존 DataStore를 유지한다.

### Round 9: UI 형태
- 기획: 사용자는 루틴을 만들 때 현재 어떤 일차를 편집 중인지 계속 알아야 한다.
- 디자인: 작은 모달보다 전체 높이 편집면이 필요하다. `1일차/2일차` 선택과 운동 목록이 함께 보여야 한다. 단, 루틴 선택 목록이 Plan 탭 본문을 밀어내면 오늘 일정 확인성이 떨어진다.
- 아키텍트: `TrainingRoute.kt`가 커져 있으므로 builder는 별도 composable로 분리한다.
- 서버: UI와 무관하지만 API request는 같은 구조를 받는다.
- 합의: Plan 탭은 현재 루틴 카드, 루틴 바꾸기, 내 루틴 만들기 진입점만 둔다. 내 루틴/기본 루틴 목록, 기본 루틴 복사, 추천 찾기는 루틴 선택 팝업 안에 둔다. 커스텀 편집은 별도 full-height editor surface로 연다.

### Round 10: 검증
- 기획: 성공 기준은 저장한 내 루틴을 시작하고 홈에서 다음 일차가 명확히 보이는 것이다.
- 디자인: 혼동 방지는 UI 회귀 테스트 없이는 검증하기 어렵다.
- 아키텍트: domain, DAO/repository, ViewModel, connected UI test 순서로 쌓아야 한다.
- 서버: API service/route 테스트와 lint/build가 필요하다.
- 합의: PR 전 앱과 서버 모두 자동 테스트를 실행하고, UI 회귀 테스트에 커스텀 루틴 핵심 흐름을 추가한다.

## 5. 핵심 사용자 흐름
1. 사용자가 Plan 탭에서 현재 루틴과 루틴 일정을 확인한다.
2. 루틴을 바꾸고 싶으면 `루틴 바꾸기`를 눌러 선택 팝업을 연다.
3. 선택 팝업에서 `내 루틴`과 `기본 루틴`을 구분해 선택하거나, 기본 루틴을 내 루틴으로 복사한다.
4. 새 루틴을 만들고 싶으면 Plan 탭 또는 선택 팝업에서 `내 루틴 만들기`를 누른다.
5. 루틴명을 입력하고 1일차 운동 부위를 펼친다.
6. 필요하면 현재 일차의 주요 부위를 선택한다. 선택하지 않으면 주요 부위는 표시되지 않고, 선택하면 추가 후보와 이미 담긴 운동이 해당 부위 기준에 맞춰진다.
7. 후보 운동의 방법이 궁금하면 `운동 방법`을 눌러 상세 팝업을 보고, 필요한 이미지는 확대해서 확인한다.
8. 운동을 추가하고 필요하면 2일차 이상을 추가한 뒤 운동 순서를 조정한다.
9. `저장`하면 팝업이 닫히고 내 루틴 목록에 남으며 현재 루틴은 바뀌지 않는다.
10. 저장한 루틴을 시작하고 싶으면 `루틴 바꾸기`에서 내 루틴을 선택한다.
11. 저장한 내 루틴을 바꾸고 싶으면 `루틴 바꾸기`의 내 루틴에서 `수정`을 눌러 같은 편집 화면으로 돌아간다.
12. 운동 기록과 일차 완료 후 기존처럼 다음 일차로 이동한다.

## 6. 수용 기준
- 1~7일차 커스텀 루틴을 만들 수 있다.
- 각 일차는 최소 1개 운동을 가져야 한다.
- 새 루틴은 기본 운동 없이 빈 일차로 시작한다.
- 새 루틴 편집 화면은 `상체 루틴` 같은 focus 라벨을 현재 일차에 따라 따라붙게 표시하지 않는다.
- 각 일차의 주요 부위는 `설정 안 함`을 기본값으로 두고, 사용자가 선택한 경우에만 저장/표시한다.
- 주요 부위 선택은 기본 상태에서 한 줄만 차지하는 드롭다운형 선택 UI로 제공한다.
- 각 일차의 주요 부위를 선택하면 운동 추가 후보는 해당 주요 부위에 맞는 운동만 표시한다.
- 주요 부위를 변경하면 이미 추가된 운동 중 새 주요 부위에 속한 운동만 남고, 벗어나는 운동은 현재 일차에서 제거된다.
- 주요 부위가 없는 커스텀 일차는 홈에서 `오늘은 전신 루틴` 같은 문구를 표시하지 않는다.
- 커스텀 루틴 카드와 홈 다음 운동 카드에는 추정 운동 시간을 표시하지 않는다.
- 내 루틴 목록의 `수정` 액션으로 저장된 커스텀 루틴을 다시 열어 수정할 수 있다.
- 현재 선택된 내 루틴의 현재 루틴 카드에서도 `수정` 액션으로 같은 편집 화면을 열 수 있다.
- 운동은 기존 카탈로그 운동만 선택 가능하다.
- 같은 일차에 이미 추가한 운동은 다시 추가할 수 없고 후보 목록에서도 보이지 않는다.
- 운동 후보는 부위별로 묶이며 기본적으로 접힌 상태로 표시된다.
- 사용자는 부위 섹션을 펼친 뒤 해당 부위 운동을 선택할 수 있다.
- 커스텀 운동 후보에서 `운동 방법`을 누르면 해당 운동 상세를 팝업으로 바로 확인할 수 있다.
- 커스텀 편집 중 열린 운동 상세에는 기록 시작 액션이 보이지 않는다.
- 운동 상세의 대표 이미지와 단계 이미지를 누르면 더 큰 이미지 뷰어가 열린다.
- `기본 루틴`과 `내 루틴` 배지가 현재 루틴 카드와 선택 목록에 표시된다.
- 4일 구성 내 루틴을 만들면 `가슴 집중`, `하체 집중`, `등 집중`, `어깨 집중`처럼 설정한 모든 일차 주요 부위가 내 루틴 카드와 현재 루틴 카드에 표시된다.
- 홈 다음 운동 카드는 `오늘 운동 예시`가 아니라 `오늘 운동 계획`을 제목으로 쓰고, 해당 일차에 들어 있는 모든 운동명을 보여준다.
- Plan 탭 본문에는 내 루틴/기본 루틴 전체 목록이 직접 펼쳐지지 않는다.
- 루틴 목록, 복사, 추천 찾기는 `루틴 바꾸기` 선택 팝업 안에서 접근한다.
- 저장만으로 현재 루틴은 변경되지 않는다.
- 저장에 성공하면 커스텀 루틴 편집 팝업은 닫힌다.
- 빌더 하단에는 `일차 삭제`와 `저장`만 정렬해 표시하고 `저장하고 시작` 버튼은 제공하지 않는다.
- 시작 시 기존 운동 로그는 삭제되지 않는다.
- 커스텀 루틴의 planned exercise id는 slot index를 포함해 기존 데이터나 import 경로에서도 기록 id가 충돌하지 않는다.
- 신규 문구는 `values`와 `values-ko`에 모두 존재한다.

## 9. 2026-05-30 운동 방법/이미지 확인 UX 메모

- 판단: 사용자는 운동을 넣기 전에 “이 운동이 맞는지” 확인해야 하며, 작은 단계 이미지는 학습용 정보로 충분하지 않다.
- 수정 방향: 후보 운동 행에는 추가와 분리된 `운동 방법` 액션을 제공한다. 이 액션은 상세 팝업을 열고, 루틴 편집 중에는 기록 시작 버튼을 숨긴다.
- 운동 방법 팝업은 기존 편집 화면을 숨기거나 다시 그리는 방식이 아니라, 현재 빌더 위에 오버레이로 떠야 한다.
- 운동 상세의 대표 이미지와 단계 이미지는 탭 시 확대 뷰어로 열어 자세 확인성을 높인다.

## 11. 2026-05-30 주요 부위/수정 UX 메모
- 피드백: 커스텀 루틴에 주요 부위 설정이 없는데도 `전신`으로 표시되고, 앱이 운동 시간을 추정해 보여주는 것이 부정확하다. 저장한 커스텀 루틴 수정도 필요하다.
- 판단: 커스텀 루틴은 사용자가 명시한 정보만 사용자에게 보여줘야 한다. 앱이 임의로 `전신`이나 `약 N분`을 붙이면 기본 루틴 처방과 내 루틴 편집 정보가 섞여 보인다.
- 수정 방향: 일차별 주요 부위 선택을 추가하되 기본값은 `설정 안 함`으로 둔다. 설정이 없으면 홈의 주요 부위 섹션, `오늘은 전신 루틴`류 헤드라인, 진행 flow를 숨긴다.
- 커스텀 루틴 카드와 홈 다음 운동 카드에서는 추정 시간을 제거하고, 내 루틴 카드에는 `수정` 액션을 제공해 같은 routine id로 다시 저장한다.

## 12. 2026-05-30 주요 부위 기반 후보 필터 UX 메모
- 피드백: 주요 부위를 `등`으로 선택했다면 운동 추가 후보도 등 운동만 나와야 한다. 이미 추가된 운동도 새 주요 부위에 속하지 않으면 남아 있으면 안 된다.
- 판단: 주요 부위 선택은 단순 라벨이 아니라 사용자의 의도를 좁히는 편집 기준이다. 선택 후에도 다른 부위 운동이 계속 보이면 사용자는 주요 부위 설정의 의미를 다시 의심한다. 다만 새 주요 부위에 속한 기존 운동까지 지우면 사용자가 이미 맞게 고른 내용을 잃는다.
- 수정 방향: `설정 안 함`은 모든 부위를 보여주고, 명확한 부위 선택은 해당 부위 후보만 보여준다. `상체`는 등/가슴/어깨/팔, `유산소/컨디셔닝`은 유산소 후보로 매핑한다.
- 주요 부위 변경 시 이미 추가된 운동은 같은 `RoutineFocus -> MuscleGroup` 기준으로 재검토한다. 새 주요 부위에 속한 운동은 유지하고, 벗어나는 운동만 현재 일차에서 제거한다. `설정 안 함`으로 되돌리는 것은 남아 있는 운동을 유지하면서 후보 범위만 전체로 넓힌다.

## 13. 2026-05-30 저장 후 닫힘 UX 메모

- 판단: 저장 성공 후 같은 편집 팝업에 남아 있으면 사용자는 저장이 완료된 다음 행동을 다시 판단해야 한다. 저장 완료는 내 루틴 목록/Plan 화면에서 이어가는 것이 더 명확하다.
- 수정 방향: 저장 실패는 기존처럼 팝업 안에서 오류를 보여주고, 저장 성공은 새 루틴 생성과 기존 루틴 수정 모두 빌더를 닫는다. 저장만으로 현재 루틴은 바꾸지 않는다.

## 14. 2026-05-30 UI 테스트 데이터 격리 메모
- 피드백: UI 테스트 후 사용자가 로컬에 저장했던 데이터가 사라졌다.
